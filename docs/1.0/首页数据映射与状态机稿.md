# 初长 V1.0｜首页数据映射与状态机稿

## 一、目标

这份文档解决 4 件事：

1. 首页时间线如何消费 `RecordItem`
2. 四个记录入口各自如何落数据
3. 进行中状态如何独立管理
4. 首页交互由哪些状态驱动

这份文档默认遵循一个原则：

**首页是“记录流的展示层”，不是业务逻辑层。**
View 只负责展示和触发事件，不直接写数据、不直接拼业务规则。

---

# 二、首页的数据来源

首页一共依赖两类数据：

## 2.1 已完成记录数据

来自 `RecordItem` 持久化数据流。

它负责：

* 渲染时间线卡片
* 提供今日记录列表
* 支持历史回看
* 为周报和成长页提供基础数据

## 2.2 进行中状态数据

来自独立的“进行中会话状态”。

它负责：

* 渲染 OngoingStateBar
* 维持睡眠中的动态计时
* App 重启后恢复正在进行的状态

---

# 三、RecordItem 首页映射原则

你给定的核心模型如下：

```swift
@Model
final class RecordItem {
    @Attribute(.unique) var id: UUID
    var timestamp: Date
    var type: String // milk, diaper, sleep, food, growth, milestone, photo
    var value: Double?
    var subType: String?
    var imageURL: String?
    var aiSummary: String?
    var tags: [String]?
    var note: String?
}
```

首页 V1 只消费以下 4 类：

* milk
* diaper
* sleep
* food

其余：

* growth
* milestone
* photo

不在记录首页首版主路径中承担高频入口，但模型可预留。

---

# 四、首页展示模型

为了避免 View 直接理解原始数据，首页应有一层展示模型：

## 4.1 TimelineDisplayItem

建议新增首页展示层模型：

```swift
struct TimelineDisplayItem: Identifiable {
    let id: UUID
    let recordID: UUID
    let timestamp: Date
    let cardStyle: TimelineCardStyle
    let leadingIcon: RecordIcon
    let title: String
    let subtitle: String?
    let imagePath: String?
    let type: RecordType
}
```

## 4.2 cardStyle

用于决定首页用哪种卡片：

```swift
enum TimelineCardStyle {
    case standard
    case foodPhoto
}
```

## 4.3 type

建议在展示层不要继续直接用裸字符串，做一层明确枚举：

```swift
enum RecordType {
    case milk
    case diaper
    case sleep
    case food
}
```

# 五、四类记录的数据映射规则

---

## 5.1 奶量记录映射

## 写入规则

当用户通过奶量 Sheet 完成记录时，创建一条 `RecordItem`：

```swift
type = "milk"
timestamp = 保存当下时间
value = 奶量数值（如 120）
subType = nil（V1 暂不强制细分）
imageURL = nil
aiSummary = nil
tags = nil
note = nil
```

## 首页映射规则

把这条记录映射为：

* `cardStyle = .standard`
* `leadingIcon = .milk`
* `title = "\(Int(value))ml"`
* `subtitle = nil`

### 示例

120ml →
首页显示：**120ml**

若未来定义奶的来源，可再扩展：

* 120ml 配方奶
* 90ml 母乳

但 V1 不建议首页先依赖未定字段。

---

## 5.2 尿布记录映射

## 写入规则

当用户点击“小便 / 大便 / 都有”之一时，创建一条 `RecordItem`：

```swift
type = "diaper"
timestamp = 保存当下时间
value = nil
subType = "pee" | "poop" | "both"
imageURL = nil
aiSummary = nil
tags = nil
note = nil
```

## 首页映射规则

* `cardStyle = .standard`
* `leadingIcon = .diaper`
* `title = 根据 subType 映射成人话`
* `subtitle = nil`

### subType 文案映射

* `pee` → `尿布：小便`
* `poop` → `尿布：大便`
* `both` → `尿布：都有`

---

## 5.3 睡眠记录映射

睡眠是首页最特殊的一类，因为它有“进行中”和“已完成”两个阶段。

## 5.3.1 睡眠开始时

**不立刻创建完成态 `RecordItem`。**

原因：

* 进行中记录不该和完成记录混在一起
* 首页正在睡眠的可见状态由 OngoingStateBar 承担
* 避免中途退出后留下半条不完整记录

所以：
点击“睡眠”开始时，写入的是**独立会话状态**，不是 `RecordItem`。

## 5.3.2 睡眠结束时

结束睡眠后才创建 `RecordItem`：

```swift
type = "sleep"
timestamp = 睡眠开始时间
value = 睡眠总时长（建议统一以秒或分钟保存）
subType = nil
imageURL = nil
aiSummary = nil
tags = nil
note = nil
```

### 关于 `timestamp`

首页与后续统计都更适合把 `timestamp` 定义为 **开始时间**。
这样：

* 时间线按事件开始发生时间排序更自然
* 详情页和趋势页都能还原“什么时候开始睡的”

### 关于 `value`

建议统一约定为：

* `sleep.value = durationInMinutes`
  或
* `sleep.value = durationInSeconds`

二选一，但必须全项目统一。
如果只考虑首页展示，分钟更直接；如果考虑未来统计精度，秒更稳。

## 首页映射规则

* `cardStyle = .standard`
* `leadingIcon = .sleep`
* `title = 睡了 x小时x分`
* `subtitle = nil`

### 文案示例

100 分钟 → `睡了 1小时40分`

---

## 5.4 辅食记录映射

辅食 V1 是“标签 + 一句话 + 可选照片”。

## 写入规则

当用户点击“完成记录”时，创建一条 `RecordItem`：

```swift
type = "food"
timestamp = 保存当下时间
value = nil
subType = nil
imageURL = 选图后的本地路径（若无图则 nil）
aiSummary = nil
tags = 选中的食材/客观标签数组
note = 用户输入的一句话（若无输入则 nil）
```

## 首页映射规则分两种

### A. 无图辅食

* `cardStyle = .standard`
* `leadingIcon = .food`
* `title = 主描述`
* `subtitle = 次描述（如果需要）`

#### 主描述生成优先级

1. 有 tags
   生成：`吃了南瓜、米粉`
2. 无 tags 但有 note
   直接使用 `note`
3. 有 tags + note
   `title = 吃了南瓜、米粉`
   `subtitle = note`

#### 示例

* tags = [南瓜, 米粉]
  → title: `吃了南瓜、米粉`
* note = “今天一直把勺子往地上丢”
  → title: `今天一直把勺子往地上丢`
* tags + note
  → title: `吃了南瓜、米粉`
  → subtitle: `今天一直把勺子往地上丢`

### B. 有图辅食

* `cardStyle = .foodPhoto`
* `leadingIcon = .food`
* `imagePath = imageURL`
* `title / subtitle` 按同样规则生成

#### 显示规则

优先展示图片，其次展示文字与时间元信息。

# 六、首页文案生成规则

为了避免 View 内写大量 if-else，建议集中做一层文案工厂：

## 6.1 TimelineContentFormatter

建议建立统一格式化器：

```swift
enum TimelineContentFormatter {
    static func makeDisplayItem(from record: RecordItem) -> TimelineDisplayItem?
}
```

它负责：

* 识别 type
* 生成 title / subtitle
* 决定 cardStyle
* 映射图标
* 返回首页展示模型

View 不直接理解：

* `subType = "pee"`
* `value = 120`
* `tags = ["南瓜", "米粉"]`

这些都由 formatter 消化。

# 七、进行中状态模型

首页首版真正需要的进行中状态只有一个：

**睡眠进行中**

建议独立为单独模型或状态存储，不与 `RecordItem` 混用。

## 7.1 建议模型：SleepSessionState

```swift
struct SleepSessionState: Codable {
    let id: UUID
    let startedAt: Date
    let isActive: Bool
}
```

如果你未来考虑多种 ongoing 状态，也可以抽象成：

```swift
enum OngoingSessionType {
    case sleep
}
```

但 V1 不建议过度抽象。

## 7.2 存储策略

进行中状态需要：

* App 重启可恢复
* 可快速读取
* 不一定需要和时间线同库复杂联动

V1 可采用：

* 本地持久存储一个轻量状态对象
* 启动时恢复
* 结束时清除

## 7.3 首页展示逻辑

如果存在 `activeSleepSession`，首页显示：

* OngoingStateBar
* 动态时长 = 当前时间 - startedAt

如果不存在：

* OngoingStateBar 隐藏

# 八、首页状态树

首页状态建议拆成三层：

## 8.1 路由状态

控制首页当前处于什么页面与模态。

```swift
struct HomeRouteState {
    var currentModule: HomeModule
    var activeSheet: ActiveSheet?
}
```

### currentModule

```swift
enum HomeModule {
    case record
    case growth
    case collection
}
```

### activeSheet

```swift
enum ActiveSheet {
    case milk
    case diaper
    case food
    case sleepControl
}
```

---

## 8.2 页面展示状态

控制首页渲染内容。

```swift
struct HomeViewState {
    var todayDisplayItems: [TimelineDisplayItem]
    var ongoingSleep: SleepSessionState?
    var undoToast: UndoToastState?
}
```

---

## 8.3 交互草稿状态

只服务各 Sheet 的临时输入，不直接持久化。

```swift
struct FoodDraftState {
    var selectedTags: [String]
    var note: String
    var selectedImagePath: String?
}
```

```swift
struct MilkDraftState {
    var selectedPreset: Int?
    var customValue: Int
    var isUsingCustomValue: Bool
}
```

尿布几乎不需要草稿态，因为点了就存。

# 九、首页事件流

下面这部分是开发实现最关键的内容：**一次点击后，到底发生什么。**

---

## 9.1 点“奶”

事件流：

1. `tapMilkEntry`
2. `activeSheet = .milk`
3. 用户点击预设值
4. 创建 `RecordItem(type: milk, value: x)`
5. 写入 SwiftData
6. 重新查询首页今日记录
7. formatter 输出新的 `TimelineDisplayItem`
8. 顶部插入标准奶量卡
9. `activeSheet = nil`
10. `undoToast = lastCreatedMilkRecord`

---

## 9.2 点“尿布”

事件流：

1. `tapDiaperEntry`
2. `activeSheet = .diaper`
3. 用户选择 pee / poop / both
4. 创建 `RecordItem(type: diaper, subType: xxx)`
5. 写入 SwiftData
6. 关闭 Sheet
7. 刷新首页时间线
8. 显示 Undo

---

## 9.3 点“睡眠”（开始）

事件流：

1. `tapSleepEntry`
2. 若 `ongoingSleep == nil`
3. 创建 `SleepSessionState(startedAt: now, isActive: true)`
4. 本地持久化 session
5. 首页显示 OngoingStateBar
6. 不创建 `RecordItem`

---

## 9.4 点“睡眠”（结束）

事件流：

1. 用户点击 OngoingStateBar 的结束按钮
2. 读取 `startedAt`
3. 计算 duration
4. 创建 `RecordItem(type: sleep, timestamp: startedAt, value: duration)`
5. 写入 SwiftData
6. 清除 `SleepSessionState`
7. OngoingStateBar 消失
8. 首页插入睡眠卡片
9. 显示 Undo

---

## 9.5 点“辅食”

事件流：

1. `tapFoodEntry`
2. `activeSheet = .food`
3. 用户选标签 / 输入 note / 选图
4. 校验完成按钮可用
5. 点击完成
6. 创建 `RecordItem(type: food, tags: ..., note: ..., imageURL: ...)`
7. 写入 SwiftData
8. 关闭 Sheet
9. formatter 决定用 `.standard` 还是 `.foodPhoto`
10. 首页插入辅食卡片
11. 显示 Undo

---

# 十、Undo 数据回滚逻辑

Undo 不是只把提示收回，而是**真正撤销刚刚写入的数据**。

## 10.1 UndoToastState

建议存：

```swift
struct UndoToastState {
    let recordID: UUID
    let message: String
}
```

## 10.2 回滚规则

当用户点击撤销时：

1. 根据 `recordID` 找到刚创建的那条 `RecordItem`
2. 从持久层删除
3. 重新生成首页 `todayDisplayItems`
4. 关闭 toast

## 10.3 睡眠特殊情况

若撤销的是刚结束的睡眠记录：

* 删除刚生成的 sleep record
* 不自动恢复 ongoing sleep session

原因：

* 结束动作已经是用户明确完成
* Undo 语义是“撤销记录”，不是“把时间倒回正在睡”

这样更清晰，也更少歧义。

# 十一、首页查询策略

首页只需要两类查询：

## 11.1 今日记录查询

查询范围：

* 今天 00:00 到当前时间
* type in [milk, diaper, sleep, food]

按 `timestamp DESC` 排序。

## 11.2 历史记录查询

当用户下拉浏览历史时，再继续拉更早日期记录。

V1 不需要在首页首屏同时混入太多历史分组逻辑。

---

# 十二、时间格式与展示规则

为了让首页像手帐，而不是后台日志，时间要轻、稳、统一。

## 12.1 卡片右侧时间

展示短时间格式：

* 10:30
* 12:05

是否显示 AM / PM，取决于全局语言和地区格式。
建议跟随系统地区设置。

## 12.2 睡眠时长格式

建议统一为：

* 小于 60 分钟：`45分钟`
* 大于等于 60 分钟：`1小时42分`

## 12.3 辅食标签拼接格式

* 最多展示前 2~3 个标签
* 超出时不在首页全展开
* 例如：

  * `吃了南瓜、米粉`
  * `吃了南瓜、米粉、香蕉`

若未来标签很多，可考虑：

* `吃了南瓜、米粉等`

但 V1 不必先做复杂截断策略。

# 十三、首页异常与边界处理

## 13.1 奶量 value 缺失

若 milk 记录缺少 value，则不展示该条，或用兜底文案：

* `记录了一次奶量`

但正常流程不应产生此类数据。

## 13.2 尿布 subType 非法

若 diaper 的 `subType` 不在预设集合内，则兜底为：

* `记录了一次尿布`

## 13.3 辅食空记录

若 food 的 tags、note、imageURL 全为空，这条记录不应允许创建。

## 13.4 图片路径失效

若辅食记录有 `imageURL` 但本地图片不存在：

* 首页退回标准文本卡
* 不因为图片丢失而让整条记录不可显示

## 13.5 睡眠状态丢失

若存在异常 session 但 startedAt 非法，首页不展示 OngoingStateBar，并清理异常状态。

# 十四、建议的首页 Store 结构

为遵守“View 不直接操作逻辑”，建议首页使用一个总的可观察状态对象。

## 14.1 HomeStore

建议承担：

* 管理首页路由状态
* 管理今日时间线数据
* 管理 ongoing sleep 状态
* 接收用户事件
* 调用 repository / service
* 产出 Undo 状态

结构可类似：

```swift
@Observable
final class HomeStore {
    var routeState: HomeRouteState
    var viewState: HomeViewState
    var foodDraft: FoodDraftState
    var milkDraft: MilkDraftState

    func onAppear()
    func handle(_ action: HomeAction)
}
```

---

# 十五、建议的 Service / Repository 分层

为了让首页不变成巨型逻辑文件，建议分成以下职责：

## Repository 层

* `RecordRepository`
* `SleepSessionRepository`

### RecordRepository

负责：

* 创建记录
* 删除记录
* 查询今日记录
* 查询历史记录

### SleepSessionRepository

负责：

* 开始睡眠
* 读取 active session
* 结束并清除 session
* 恢复 session

## Formatter 层

* `TimelineContentFormatter`

负责把 `RecordItem` 转成 `TimelineDisplayItem`

## Store 层

* `HomeStore`

负责把用户动作串起来。

# 十六、首页开发落地顺序

我建议按这个顺序开发，返工最少：

## 第 1 步

搭首页壳层：

* 顶部导航
* ScrollView
* 底部胶囊
* OngoingStateBar 占位
* UndoToast 占位

## 第 2 步

接 `RecordRepository` 与 `TimelineContentFormatter`

* 跑通标准卡片渲染
* 先显示奶 / 尿布 / 睡眠 / 辅食的静态假数据

## 第 3 步

跑通奶量 Sheet

* 预设值保存
* 时间线插入
* Undo

## 第 4 步

跑通尿布 Sheet

* 一击保存
* Undo

## 第 5 步

跑通睡眠 ongoing session

* 开始
* 状态条展示
* 结束
* 生成 sleep 卡片

## 第 6 步

跑通辅食 Sheet

* 标签
* note
* 可选照片
* 文本卡 / 图片卡两种渲染
