# 1. 文档目标

把“首页记录编辑 / 删除”这次开发工作，从单线任务清单进一步拆成可以并行执行的多线程开发方案。

本稿解决以下问题：

1. 最适合拆成几条开发线程
2. 每条线程负责哪些文件和模块
3. 哪些接口必须先冻结再并行
4. 哪些热点文件必须单 owner
5. 集成节奏如何安排，才能在提高并行度的同时控制质量

---

# 2. 总体结论

本次开发建议拆成：

1. `5 条功能开发线程`
2. `1 条持续集成 / QA 线程`

不建议拆成更多功能线程，原因如下：

1. 核心热点文件集中在 `HomeStore`、`HomeModels`、`HomeAction`、`MainTabContentView`、`RecordRepository`
2. 继续细拆会让冲突从“代码实现”转移到“合并协调”
3. 编辑与删除是强状态型需求，接口不冻结就并行，会明显拉低质量

---

# 3. 线程总览

## 线程 A：核心状态与路由

负责首页记录编辑删除的状态域、路由、Action 与主流程骨架。

## 线程 B：时间线交互 UI

负责时间线 Cell 点按、长按、菜单入口、删除确认入口与卡片反馈。

## 线程 C：数据变更与恢复

负责 Repository 更新、删除恢复快照、图片清理策略与持久化一致性。

## 线程 D：编辑器 A

负责奶量与尿布两类记录的编辑态复用、预填和保存。

## 线程 E：编辑器 B

负责睡眠与辅食两类记录的编辑态复用、预填和保存。

## 线程 F：持续集成与 QA

负责契约冻结、集成验证、回归测试、冲突检查和验收收口。

---

# 4. 为什么是这 6 条线

## 4.1 能最大化并行

这 6 条线之间只有少量强依赖：

1. 线程 A 先冻结状态与接口
2. 线程 B、C、D、E 在冻结接口上并行
3. 线程 F 持续跟进，不阻塞主开发

## 4.2 能控制热点冲突

最容易出冲突的文件不超过 5 个：

1. `sprout/Features/Home/HomeStore.swift`
2. `sprout/Features/Home/HomeModels.swift`
3. `sprout/Features/Home/HomeAction.swift`
4. `sprout/Features/Shell/MainTabContentView.swift`
5. `sprout/Domain/Records/RecordRepository.swift`

这些文件必须提前指定单 owner 或明确最后集成 owner。

## 4.3 能维持质量

把高耦合状态逻辑和低耦合 UI 复用拆开后：

1. 状态机不会被多人同时写坏
2. 四类编辑页可以分两条线独立推进
3. Repository 与测试可以提前建立质量兜底

---

# 5. 线程 Owner 规则

每条线程必须有明确 owner。

## 5.1 单 owner 原则

一个热点文件在一个阶段只能有一个 owner。

## 5.2 非 owner 修改原则

其他线程若确实需要改热点文件：

1. 先提需求给 owner
2. 由 owner 合并
3. 不允许多人直接抢改

## 5.3 推荐 owner 分配

1. 线程 A owner：`HomeStore / HomeModels / HomeAction / MainTabContentView`
2. 线程 C owner：`RecordRepository / RecordItem / RecordValidator`
3. 线程 B owner：时间线卡片与交互组件
4. 线程 D owner：奶量、尿布编辑页
5. 线程 E owner：睡眠、辅食编辑页
6. 线程 F owner：测试、构建、联调、验收清单

---

# 6. 接口冻结清单

在所有功能线程正式并行之前，线程 A 必须先冻结以下契约。

## 6.1 状态契约

至少冻结：

1. `RecordCellInteractionState`
2. `RecordEditorRouteState`
3. `RecordDeleteState`
4. `RecordFeedbackState`
5. `RecordMutationState`

## 6.2 Action 契约

至少冻结：

1. `tapTimelineRecord(UUID)`
2. `longPressTimelineRecord(UUID)`
3. `releaseTimelineRecordPress`
4. `selectRecordContextEdit(UUID)`
5. `selectRecordContextDelete(UUID)`
6. `cancelDeleteRecord`
7. `confirmDeleteRecord`
8. `dismissRecordEditor`
9. `saveRecordEdits`
10. `undoDeletedRecord`

## 6.3 路由契约

至少冻结：

1. `RecordEditorType`
2. `RecordEditorMode`
3. `RecordSheetRoute` 或 `RecordEditorRouteState`

## 6.4 Repository 契约

至少冻结：

1. `fetchRecord(id:)`
2. `updateFeedingRecord(...)`
3. `updateDiaperRecord(...)`
4. `updateSleepRecord(...)`
5. `updateFoodRecord(...)`
6. `deleteRecord(...)`
7. `restoreDeletedRecord(...)`

# 8. 并行开发阶段建议

建议拆成 4 个阶段推进。

## Phase 0：契约冻结

由线程 A 和线程 C 先完成。

目标：

1. 冻结状态与事件
2. 冻结路由结构
3. 冻结 Repository 更新接口
4. 明确删除快照策略

建议时长：

1. `0.5 - 1 天`

## Phase 1：主骨架并行实现

由线程 B、C、D、E 并行推进，线程 A 负责接口答疑和少量承接。

目标：

1. 时间线交互层能触发主流程
2. Repository 有编辑 / 删除 / 恢复能力
3. 四类编辑页可进入编辑模式

建议时长：

1. `2 - 3 天`

## Phase 2：联调与语义统一

由线程 A 主导，线程 B、C、D、E 配合。

目标：

1. 把编辑保存、删除确认、撤销恢复全部串起来
2. 统一 Toast 语义
3. 打通错误处理与边界情况

建议时长：

1. `1 - 1.5 天`

## Phase 3：回归与稳定性收口

由线程 F 主导。

目标：

1. 编译通过
2. 单元测试通过
3. 主路径手工验收通过
4. 深色模式和滚动手势无明显回归

建议时长：

1. `1 - 1.5 天`

# 9. 线程 A：核心状态与路由

## 目标

建立所有线程共享的主流程骨架，避免后续实现各自为政。

## 负责文件

1. `sprout/Features/Home/HomeStore.swift`
2. `sprout/Features/Home/HomeModels.swift`
3. `sprout/Features/Home/HomeAction.swift`
4. `sprout/Features/Shell/MainTabContentView.swift`

## 任务清单

### A-01 新增记录编辑删除状态域

**优先级**：P0
**目标**：把编辑删除流程所需状态正式放进首页状态树。

**验收标准**

1. 新增状态域命名清晰
2. 不污染已有创建记录状态
3. 状态与前述状态机文档一致

### A-02 新增首页记录管理 Action

**优先级**：P0
**目标**：让时间线交互流程拥有明确事件入口。

**验收标准**

1. Action 命名统一
2. 不与底部胶囊新增记录 Action 混淆
3. 可以覆盖点按、长按、编辑保存、删除确认、删除恢复

### A-03 建立记录编辑路由

**优先级**：P0
**目标**：统一新增与编辑的 Sheet 路由语义。

**验收标准**

1. 四类编辑页支持 `create` / `edit`
2. `.sheet(item:)` 接线清晰
3. 不出现 `editMilk` 这类路由爆炸

### A-04 接入主交互焦点控制

**优先级**：P0
**目标**：保证任一时刻只有一个主焦点。

**验收标准**

1. 编辑与删除确认不会并存
2. 菜单关闭后无残留状态
3. Toast 不会与编辑 Sheet 打架

### A-05 统一反馈层映射

**优先级**：P1
**目标**：统一 `message`、`undo create`、`undo delete` 三种反馈语义。

**验收标准**

1. 新增记录撤销逻辑不被破坏
2. 删除记录撤销可独立运行
3. `MainTabContentView` 的 overlay 逻辑保持可读

# 14. 线程 F：持续集成与 QA

## 目标

把并行开发带来的合并风险提前暴露，而不是最后集中爆炸。

## 负责内容

1. 编译检查
2. 测试补齐与回归
3. UI 主路径验收
4. 多线程改动的每日集成

## 任务清单

### F-01 契约冻结检查

**优先级**：P0
**目标**：确认线程 A / C 输出的契约不再随意改动。

**验收标准**

1. 状态名、Action 名、接口名已经固定
2. 其他线程已切换到统一契约

### F-02 每日两次集成

**优先级**：P0
**目标**：减少分支长期漂移。

**建议节奏**

1. 午间一次
2. 收工前一次

**验收标准**

1. 每次集成后主分支可编译
2. 冲突当日解决，不积压

### F-03 主路径回归测试

**优先级**：P0
**目标**：覆盖编辑、删除、恢复三条主路径。

**验收标准**

1. 点按进入编辑可用
2. 长按删除确认可用
3. 删除撤销可用
4. 改时间后排序正确

### F-04 边界回归

**优先级**：P1
**目标**：覆盖最容易漏掉的异常路径。

**验收标准**

1. 删除不存在记录不会崩溃
2. 编辑不存在记录有轻提示
3. 连续删除只保留最近一次撤销
4. 深色模式下页面仍可读

### F-05 发布前验收清单

**优先级**：P1
**目标**：形成最终 merge gate。

**验收标准**

1. 所有 P0 完成
2. 主路径人工验收通过
3. 自动化测试通过
4. 无已知 blocker

# 16. 不建议的拆法

以下拆法看起来更细，但实际上会降低效率。

## 16.1 把每一种记录类型都拆成单独线程

问题：

1. 路由和状态仍集中在 `HomeStore`
2. 四类编辑页会反复修改共享逻辑
3. 合并成本高于并行收益

## 16.2 把 `HomeStore` 交给多人共同修改

问题：

1. 很容易出现状态覆盖
2. 事件命名失控
3. 最后联调难度陡增

## 16.3 把测试放到最后统一补

问题：

1. 缺陷暴露太晚
2. 回归成本太高
3. 会让“看起来做完了”但其实不可合并

2. 修复午间集成问题

## 收工前集成

1. 再合并一次
2. 确保主分支可编译
3. 更新 blocker 列表

# 18. 质量红线

无论如何并行，以下红线不能破。

1. 不允许 `HomeStore` 出现多条未收敛并行分支长期漂移
2. 不允许删除动作无确认直接落地
3. 不允许新增记录撤销被删除撤销逻辑破坏
4. 不允许编辑保存后整页闪烁重载
5. 不允许用亮红色破坏既定视觉气质
6. 不允许跳过数据层测试直接合并 Repository 改动

# 20. 最终验收口径

如果并行开发完成，最终至少要满足：

1. 点按任意记录可进入对应编辑页
2. 长按任意记录可进入菜单
3. 菜单中只有 `编辑记录` 与 `删除记录`
4. 删除必须经过确认
5. 删除成功后可以撤销
6. 奶量、尿布、睡眠、辅食四类记录都可编辑
7. 改时间后时间线排序正确
8. 新增记录撤销与删除记录撤销不会互相污染
9. 深色模式下所有状态可读
10. 主分支始终可编译、可回归
