# 1. 文档目标

把首页记录 Cell 的编辑与删除交互，从“页面与流程定义”推进到“可以直接拆状态与事件实现”的层面。

本稿解决以下问题：

1. 首页记录 Cell 编辑删除能力由哪些状态域组成
2. 每个状态由谁持有、何时切换
3. 用户动作如何映射为事件
4. 哪些事件需要守卫条件
5. 与现有 `HomeStore`、`HomeAction`、`ActiveSheet` 结构如何衔接

本稿不解决：

1. 最终 UI 视觉细节
2. Repository 的底层持久化代码
3. 具体测试代码实现

---

# 2. 状态机设计原则

首页记录编辑删除状态机必须遵循以下原则：

## 2.1 内容优先

默认状态永远是时间线浏览，而不是管理模式。

## 2.2 单焦点

任意时刻只允许一个主交互焦点存在：

1. 要么浏览时间线
2. 要么展示 Context Menu
3. 要么展示删除确认
4. 要么展示编辑 Sheet

## 2.3 结果反馈后置

结果 Toast 只能在主交互完成后出现，不能与编辑 Sheet 或删除确认层叠加。

## 2.4 状态解耦

以下状态必须拆开管理：

1. 页面数据状态
2. 交互焦点状态
3. 编辑路由状态
4. 删除确认状态
5. 结果反馈状态

## 2.5 最小可恢复

删除必须支持短时撤销，但 V1 只维护最近一条删除的恢复能力。

# 3. 当前实现基础

当前首页已有以下基础能力：

1. `HomeRouteState.activeSheet`
2. `HomeViewState.undoToast`
3. `HomeViewState.messageToast`
4. `HomeAction.dismissSheet`
5. `HomeAction.undoLastRecord`
6. `RecordRepository.deleteRecord(id:)`

当前首页缺少以下能力：

1. 记录 Cell 点按进入编辑的事件
2. 记录 Cell 长按管理事件
3. 记录删除确认状态
4. 记录编辑上下文状态
5. 记录更新接口
6. 删除后针对“历史删除”与“刚创建撤销”的语义区分

因此，本稿将明确新增状态域与新增事件，并尽量复用现有 Toast / Sheet 基础设施。

---

# 4. 状态域总览

首页记录编辑删除能力建议拆成 5 个状态域：

1. `RecordCellInteractionState`
2. `RecordEditorRouteState`
3. `RecordDeleteState`
4. `RecordFeedbackState`
5. `RecordMutationState`

它们的职责分别如下：

## 4.1 RecordCellInteractionState

负责描述“当前是否正在针对某条 Cell 进行交互”。

## 4.2 RecordEditorRouteState

负责描述“当前是否正在编辑哪条记录，以及编辑的是哪种记录类型”。

## 4.3 RecordDeleteState

负责描述“当前是否正在确认删除哪条记录”。

## 4.4 RecordFeedbackState

负责描述“当前底部反馈层展示的是哪一种结果提示”。

## 4.5 RecordMutationState

负责描述“当前是否正在提交一次编辑或删除动作”。

# 5. 核心状态定义

---

## 5.1 RecordCellInteractionState

```swift
enum RecordCellInteractionState: Equatable {
    case idle
    case pressing(recordID: UUID)
    case menuTargeted(recordID: UUID)
}
```

### 说明

* `idle`
  页面处于默认浏览状态

* `pressing(recordID:)`
  用户已点中某条记录，正在触发瞬时反馈

* `menuTargeted(recordID:)`
  用户长按某条记录后，该条记录成为菜单上下文目标

### 使用边界

1. `pressing` 是瞬时态，不应长时间持有
2. `menuTargeted` 只用于菜单上下文，不应承担删除或编辑逻辑本身

---

## 5.2 RecordEditorRouteState

建议不要继续在 `ActiveSheet` 里增加大量 `.editMilk / .editDiaper / .editFood / .editSleep` 枚举，而是拆成“记录 Sheet 路由 + 模式”。

```swift
enum RecordEditorType: Equatable {
    case milk
    case diaper
    case sleep
    case food
}

enum RecordEditorMode: Equatable {
    case create
    case edit(recordID: UUID)
}

struct RecordEditorRouteState: Equatable {
    let editorType: RecordEditorType
    let mode: RecordEditorMode
}
```

### 说明

* `editorType`
  决定进入哪一类编辑页

* `mode`
  决定该页是“新增”还是“编辑”

### 原因

1. 避免 `ActiveSheet` 爆炸式增长
2. 让四类记录在新增与编辑场景下复用同一页面
3. 让 `HomeStore` 更清晰地区分“入口创建”和“时间线编辑”

---

## 5.3 RecordDeleteState

```swift
enum RecordDeleteState: Equatable {
    case idle
    case confirming(summary: RecordDeleteSummary)
}

struct RecordDeleteSummary: Equatable {
    let recordID: UUID
    let title: String
    let subtitle: String?
    let timestamp: Date
    let type: RecordType
}
```

### 说明

* `idle`
  当前没有删除确认流程

* `confirming`
  当前正展示删除确认 Sheet

### 设计意图

确认层展示所需信息应在状态中准备好，而不是删除确认弹出时再临时从多个对象拼装。

---

## 5.4 RecordFeedbackState

建议把“底部结果反馈”的语义从 `undoToast` 和 `messageToast` 再抽象一层。

```swift
enum RecordFeedbackState: Equatable {
    case none
    case message(String)
    case undoDelete(DeletedRecordSnapshot)
    case undoCreate(recordID: UUID, message: String)
}
```

### 说明

* `none`
  无反馈

* `message`
  普通轻提示，例如“已更新记录”

* `undoDelete`
  删除后的可撤销反馈

* `undoCreate`
  沿用当前新增记录后的撤销反馈

### 设计意图

当前首页已存在 `UndoToastState`，但它只存 `recordID + message`，不足以区分：

1. 这是删除撤销还是创建撤销
2. 删除撤销时需要恢复哪些字段

因此建议在逻辑层先区分反馈语义，再映射到 UI Toast。

---

## 5.5 RecordMutationState

```swift
enum RecordMutationState: Equatable {
    case idle
    case savingEdit(recordID: UUID)
    case deleting(recordID: UUID)
    case restoringDeleted(recordID: UUID)
}
```

### 说明

该状态用于：

1. 防止重复提交
2. 防止多个主流程重叠
3. 给后续按钮禁用、loading 或调试日志提供依据

---

# 6. 删除恢复快照

删除撤销与创建撤销不同。

创建撤销只需要删除掉刚创建的记录即可。  
删除撤销则需要重新插入被删记录，因此 V1 需要保留一份最小恢复快照。

建议定义：

```swift
struct DeletedRecordSnapshot: Equatable {
    let id: UUID
    let timestamp: Date
    let type: String
    let value: Double?
    let leftNursingSeconds: Int
    let rightNursingSeconds: Int
    let subType: String?
    let imageURL: String?
    let aiSummary: String?
    let tags: [String]?
    let note: String?
    let message: String
}
```

### 说明

1. 该快照应在真正删除前生成
2. 若带图片记录被删除，撤销时必须基于可恢复策略处理图片路径
3. V1 若删除时同步删本地图片，需额外定义恢复策略；更稳的做法是把图片清理延后到撤销窗口失效后执行

# 7. 页面主交互状态机

首页记录 Cell 编辑删除能力的主交互焦点可抽象为：

```swift
enum RecordInteractionFocusState: Equatable {
    case timelineIdle
    case recordPressed(UUID)
    case contextMenu(UUID)
    case deleteConfirming(UUID)
    case editing(UUID)
}
```

## 7.1 状态说明

* `timelineIdle`
  默认时间线浏览状态

* `recordPressed`
  点按瞬时反馈状态

* `contextMenu`
  长按菜单焦点状态

* `deleteConfirming`
  删除确认焦点状态

* `editing`
  编辑 Sheet 焦点状态

## 7.2 主状态流

```text
timelineIdle
  ├─ tapRecord → recordPressed → editing → timelineIdle
  └─ longPressRecord → contextMenu

contextMenu
  ├─ selectEdit → editing
  ├─ selectDelete → deleteConfirming
  └─ dismissMenu → timelineIdle

deleteConfirming
  ├─ confirmDelete → timelineIdle + undoDeleteFeedback
  └─ cancelDelete → timelineIdle
```

## 7.3 关键约束

1. 同一时刻只能存在一个 `focus state`
2. `editing` 与 `deleteConfirming` 不允许并存
3. `recordPressed` 必须自动进入下一状态或回到 `timelineIdle`

# 8. 事件定义

事件分为 4 类：

1. 用户手势事件
2. 菜单与 Sheet 事件
3. 数据写入结果事件
4. 反馈层事件

---

## 8.1 用户手势事件

```swift
enum HomeRecordInteractionEvent: Equatable {
    case tapRecord(UUID)
    case longPressRecord(UUID)
    case releaseRecordPress
    case dismissContextMenu
}
```

### 说明

* `tapRecord`
  用户点按一条时间线记录

* `longPressRecord`
  用户长按一条时间线记录

* `releaseRecordPress`
  点按反馈结束，准备进入编辑

* `dismissContextMenu`
  用户取消菜单或菜单自然消失

---

## 8.2 菜单与 Sheet 事件

```swift
enum HomeRecordManagementEvent: Equatable {
    case selectContextEdit(UUID)
    case selectContextDelete(UUID)
    case cancelDeleteConfirmation
    case confirmDelete
    case dismissEditor
    case saveEditorChanges
}
```

### 说明

* `selectContextEdit`
  从菜单进入编辑

* `selectContextDelete`
  从菜单进入删除确认

* `cancelDeleteConfirmation`
  取消删除

* `confirmDelete`
  确认删除

* `dismissEditor`
  主动关闭编辑页

* `saveEditorChanges`
  点击“保存修改”

---

## 8.3 数据结果事件

```swift
enum HomeRecordMutationEvent: Equatable {
    case editSaveSucceeded(recordID: UUID)
    case editSaveFailed(message: String)
    case deleteSucceeded(snapshot: DeletedRecordSnapshot)
    case deleteFailed(message: String)
    case restoreDeletedSucceeded(recordID: UUID)
    case restoreDeletedFailed(message: String)
}
```

### 说明

这些事件由 Repository / Store 异步结果驱动，不直接由 View 发起。

---

## 8.4 反馈层事件

```swift
enum HomeRecordFeedbackEvent: Equatable {
    case tapUndoDelete
    case tapUndoCreate
    case dismissUndoFeedback
    case dismissMessageFeedback
    case feedbackTimeout
}
```

# 9. 事件表

---

## 9.1 点按进入编辑事件表

| 事件 | 前置状态 | 守卫条件 | 状态变化 | 结果 |
| --- | --- | --- | --- | --- |
| `tapRecord(recordID)` | `timelineIdle` | 当前无 Sheet、无删除确认、无写入中状态 | `focus: timelineIdle -> recordPressed(recordID)` | Cell 出现瞬时按压反馈 |
| `releaseRecordPress` | `recordPressed(recordID)` | 该记录仍存在 | `focus -> editing(recordID)`，`editorRoute = edit(recordID)` | 打开对应记录类型编辑页 |
| `releaseRecordPress` | `recordPressed(recordID)` | 记录已不存在 | `focus -> timelineIdle`，`feedback = message("这条记录已不存在。")` | 不打开编辑页 |

---

## 9.2 长按菜单事件表

| 事件 | 前置状态 | 守卫条件 | 状态变化 | 结果 |
| --- | --- | --- | --- | --- |
| `longPressRecord(recordID)` | `timelineIdle` | 当前未滚动、无 Sheet、无删除确认 | `interaction = menuTargeted(recordID)`，`focus = contextMenu(recordID)` | 打开菜单 |
| `dismissContextMenu` | `contextMenu(recordID)` | 无 | `interaction -> idle`，`focus -> timelineIdle` | 关闭菜单 |
| `selectContextEdit(recordID)` | `contextMenu(recordID)` | 记录存在 | `interaction -> idle`，`focus -> editing(recordID)`，`editorRoute = edit(recordID)` | 进入编辑 |
| `selectContextDelete(recordID)` | `contextMenu(recordID)` | 记录存在 | `interaction -> idle`，`deleteState = confirming(summary)`，`focus -> deleteConfirming(recordID)` | 打开删除确认 |

---

## 9.3 删除确认事件表

| 事件 | 前置状态 | 守卫条件 | 状态变化 | 结果 |
| --- | --- | --- | --- | --- |
| `cancelDeleteConfirmation` | `deleteConfirming(recordID)` | 无 | `deleteState -> idle`，`focus -> timelineIdle` | 关闭删除确认 |
| `confirmDelete` | `deleteConfirming(recordID)` | `mutation == idle` | `mutation -> deleting(recordID)` | 开始执行删除 |
| `deleteSucceeded(snapshot)` | `mutation == deleting(recordID)` | 无 | `deleteState -> idle`，`focus -> timelineIdle`，`mutation -> idle`，`feedback -> undoDelete(snapshot)` | 列表移除该记录并出现撤销 |
| `deleteFailed(message)` | `mutation == deleting(recordID)` | 无 | `deleteState -> idle`，`focus -> timelineIdle`，`mutation -> idle`，`feedback -> message(message)` | 显示删除失败提示 |

---

## 9.4 编辑保存事件表

| 事件 | 前置状态 | 守卫条件 | 状态变化 | 结果 |
| --- | --- | --- | --- | --- |
| `saveEditorChanges` | `editing(recordID)` | 表单合法，`mutation == idle` | `mutation -> savingEdit(recordID)` | 开始保存编辑 |
| `editSaveSucceeded(recordID)` | `mutation == savingEdit(recordID)` | 无 | `editorRoute = nil`，`focus -> timelineIdle`，`mutation -> idle`，`feedback -> message("已更新记录")` | 关闭编辑页并刷新时间线 |
| `editSaveFailed(message)` | `mutation == savingEdit(recordID)` | 无 | `mutation -> idle`，`feedback -> message(message)` | 保持编辑页，提示错误 |
| `dismissEditor` | `editing(recordID)` | 当前未保存中，或允许用户主动关闭 | `editorRoute = nil`，`focus -> timelineIdle` | 关闭编辑页 |

---

## 9.5 反馈层事件表

| 事件 | 前置状态 | 守卫条件 | 状态变化 | 结果 |
| --- | --- | --- | --- | --- |
| `tapUndoDelete` | `feedback == undoDelete(snapshot)` | `mutation == idle` | `mutation -> restoringDeleted(snapshot.id)` | 开始恢复记录 |
| `restoreDeletedSucceeded(recordID)` | `mutation == restoringDeleted(recordID)` | 无 | `mutation -> idle`，`feedback -> message("已恢复记录")` | 记录恢复并刷新时间线 |
| `restoreDeletedFailed(message)` | `mutation == restoringDeleted(recordID)` | 无 | `mutation -> idle`，`feedback -> message(message)` | 提示恢复失败 |
| `tapUndoCreate` | `feedback == undoCreate(recordID, _)` | `mutation == idle` | 沿用当前新增撤销逻辑 | 删除刚创建记录 |
| `dismissUndoFeedback` | `feedback == undoDelete / undoCreate` | 无 | `feedback -> none` | 关闭 Undo Toast |
| `dismissMessageFeedback` | `feedback == message(_)` | 无 | `feedback -> none` | 关闭轻提示 |
| `feedbackTimeout` | 任一反馈态 | 无 | `feedback -> none` | 到时自动消失 |

# 10. 守卫条件

以下守卫条件必须明确，否则实现时容易出现状态串扰。

## 10.1 canOpenRecordEditor

允许进入编辑页的条件：

1. 当前无其他编辑 Sheet
2. 当前无删除确认 Sheet
3. 当前不在保存中或删除中
4. 目标记录仍存在

## 10.2 canOpenDeleteConfirmation

允许进入删除确认的条件：

1. 当前无编辑 Sheet
2. 当前无其他删除确认
3. 当前不在写入中
4. 目标记录仍存在

## 10.3 canMutateRecord

允许执行编辑保存或删除的条件：

1. `mutationState == idle`
2. 当前上下文中的目标记录与事件记录一致

## 10.4 canUndoDelete

允许撤销删除的条件：

1. 当前反馈态为 `undoDelete`
2. 还处于撤销窗口内
3. 当前不在其他写入流程中

# 11. 与现有 HomeStore 的映射建议

---

## 11.1 现有结构问题

当前首页新增记录流程主要依赖：

1. `HomeRouteState.activeSheet`
2. `HomeViewState.undoToast`
3. `HomeViewState.messageToast`
4. `FoodDraftState`
5. `FeedingDraftState`

这套结构足以支撑“创建记录”，但不足以支撑“时间线记录编辑删除”，原因包括：

1. 创建流程的入口来自底部胶囊
2. 编辑流程的入口来自时间线 Cell
3. 删除流程需要确认态和恢复快照
4. 新增与编辑共用 Sheet，但上下文不同

---

## 11.2 建议新增状态

建议在 `HomeRouteState` 或 `HomeViewState` 中新增以下字段：

```swift
struct HomeRouteState {
    var activeSheet: ActiveSheet?
    var recordEditorRoute: RecordEditorRouteState?
}

struct HomeViewState {
    var recordInteraction: RecordCellInteractionState = .idle
    var recordDeleteState: RecordDeleteState = .idle
    var recordFeedbackState: RecordFeedbackState = .none
    var recordMutationState: RecordMutationState = .idle
}
```

### 说明

若希望减少 Route / ViewState 分散，也可把 `recordEditorRoute` 放到 `HomeViewState` 中。  
核心要求不是放哪里，而是：

1. 编辑路由必须独立于新增 Sheet 路由
2. 删除确认必须有独立状态
3. 反馈语义必须区分删除撤销与创建撤销

---

## 11.3 建议新增 Action

当前 `HomeAction` 需要新增至少以下事件：

```swift
enum HomeAction {
    case tapTimelineRecord(UUID)
    case longPressTimelineRecord(UUID)
    case releaseTimelineRecordPress
    case dismissRecordContextMenu
    case selectRecordContextEdit(UUID)
    case selectRecordContextDelete(UUID)
    case cancelDeleteRecord
    case confirmDeleteRecord
    case dismissRecordEditor
    case saveRecordEdits
    case undoDeletedRecord
}
```

### 说明

这些 Action 与现有底部胶囊的新增记录 Action 不应混用。

---

# 12. 编辑路由建议

为了不污染当前 `ActiveSheet`，建议将首页记录编辑统一收敛为：

```swift
enum RecordSheetRoute: Equatable, Identifiable {
    case create(RecordEditorType)
    case edit(RecordEditorType, UUID)

    var id: String { ... }
}
```

然后让首页统一对接：

```swift
var recordSheetRoute: RecordSheetRoute?
```

## 好处

1. 创建与编辑共用一条路由
2. 四种类型无需重复维护两套 sheet 体系
3. `MainTabContentView` 的 `.sheet(item:)` 映射更清晰

---

# 13. 记录存在性校验

编辑与删除都必须在正式提交前再次校验目标记录是否存在。

## 13.1 校验时机

1. 点按进入编辑前
2. 从菜单选择编辑前
3. 从菜单选择删除前
4. 点击保存修改前
5. 点击确认删除前

## 13.2 校验失败结果

统一结果：

1. 关闭当前中间态
2. 回到 `timelineIdle`
3. 显示 `这条记录已不存在。`

---

# 14. 时间线刷新规则

编辑与删除成功后，时间线必须刷新，但刷新策略要避免用户感知为“整页重载”。

## 14.1 编辑成功后

1. 刷新 today 与 history 数据
2. 若时间变化，按新时间重排
3. 保持当前滚动位置尽量稳定
4. 再展示 `已更新记录`

## 14.2 删除成功后

1. 从时间线移除目标记录
2. 记录删除快照
3. 展示 `已删除记录` Undo Toast

## 14.3 删除恢复后

1. 重新插入记录
2. 按时间排序
3. 展示 `已恢复记录`

# 15. 边界情况

---

## 15.1 连续删除多条记录

V1 规则：

1. 允许连续删除
2. 仅保留最近一条的删除恢复快照
3. 新的删除反馈覆盖旧的删除反馈

---

## 15.2 编辑中收到其他反馈

当编辑 Sheet 打开时：

1. 不展示新的 Undo Toast
2. 不展示新的 Message Toast
3. 若后台有反馈事件，应在编辑结束后再决定是否展示

---

## 15.3 删除确认与编辑冲突

如果已经打开删除确认：

1. 不允许再打开编辑页
2. 不允许再响应其他 Cell 的点按与长按

如果已经进入编辑：

1. 不允许再打开删除确认
2. 不允许其他 Cell 再进入菜单态

---

## 15.4 正在滚动时长按

若首页滚动意图明确为 scrolling：

1. 忽略 `longPressRecord`
2. 待滚动停止后才可再次进入菜单态

---

## 15.5 睡眠记录的特殊性

睡眠记录编辑必须处理：

1. 开始时间
2. 结束时间
3. 自动重算时长

保存时若校验失败：

1. 保持编辑态
2. 在编辑页内提示错误
3. 不关闭 Sheet

# 17. 验收标准

## 17.1 状态焦点

1. 任一时刻只有一个主交互焦点
2. 编辑与删除确认不会同时出现
3. 菜单关闭后不会残留选中态

## 17.2 删除流程

1. 长按记录可以进入删除确认
2. 确认删除后记录移除
3. 删除成功后出现撤销反馈
4. 撤销后记录恢复

## 17.3 编辑流程

1. 点按记录可以进入编辑
2. 编辑页能正确加载原始数据
3. 保存后记录更新
4. 改时间后列表正确重排

## 17.4 边界稳定性

1. 删除不存在记录不会崩溃
2. 编辑不存在记录会回到空闲态并提示
3. 连续删除时只保留最近一条撤销
4. 反馈层不会与编辑页打架
