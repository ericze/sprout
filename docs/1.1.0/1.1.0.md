# Auth + Supabase Backend + Cloud Sync 实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 基于 [auth-supabase-sync-design.md](../specs/2026-04-09-auth-supabase-sync-design.md) 落地 Phase 2：邮箱登录、Supabase 后端、单账号多设备云同步、图片备份，以及与现有本地优先数据流兼容的删除/撤销语义。

**Architecture:** 继续保持 local-first。`AuthManager` 负责身份与账号绑定，`SyncEngine` 负责 push/pull 编排，`SupabaseService` 负责 Supabase SDK 封装，Repository 继续拥有本地创建/编辑/删除语义，但在写路径中补充 dirty 标记与 deletion tombstone。云同步在 Phase 2 对所有已登录用户开放，不做客户端假权限边界。

**Tech Stack:** SwiftUI, SwiftData, Observation, StoreKit 2（已有）, Supabase Swift SDK, XCTest / Swift Testing

**Release**: 1.1.0

**Spec:** `docs/superpowers/specs/2026-04-09-auth-supabase-sync-design.md`

**Implementation Constraints:**
- 只允许一个本地 SwiftData store 绑定一个 Supabase 用户。
- 不允许用设备时间和服务器时间直接做冲突决策。
- 删除必须继续兼容当前 4 秒 undo 语义。
- `MemoryEntry` 必须保留多图能力。
- `Cloud Sync` 在 Phase 2 是登录可用，不做 Pro gate。
- 变更尽量保持最小 diff，先接入新字段，不重命名已有本地字段。

**Plan Outcome:** 交付 1 份可执行 SQL、1 组本地迁移与同步基础设施、2 个用户界面（`AccountView` / `CloudSyncView`）、一组单元测试与一组 Supabase 集成测试。

## 建议执行顺序

1. PR1: 配置 + service 骨架
2. PR2: migration + bootstrap
3. PR3: auth + account binding
4. PR4: repository dirty/tombstone
5. PR5: sync push
6. PR6: sync pull + assets + weekly letter recompute
7. PR7: UI integration
8. PR8: tests + hardening + manual QA

这个顺序的原因：
- 先把外部依赖和边界层搭起来，避免后续同步逻辑直接绑 SDK。
